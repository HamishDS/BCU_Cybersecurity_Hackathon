# CTU IoT Labeled Network Log Headers (Zeek/Bro conn.log)

This repository includes labeled network flow logs derived from Zeek (formerly Bro) `conn.log` outputs. Below is a concise explanation of each header observed in the CSV files, presented in a table with what each field represents and why it’s useful in analysis or detection.

Note: The fields are based on Zeek `conn.log` conventions and the CTU IoT Malware Capture labeled datasets. Field names appear without dots in the CSV header due to formatting, but conceptually map to Zeek’s dotted names (e.g., `id.orig_h`).

| Header | Description (What) | Purpose (Why) |
|---|---|---|
| ts | Timestamp (epoch seconds) when Zeek first saw the connection. | Provides temporal context, enables time-window analytics, sequence reconstruction, and correlation with other logs. |
| uid | Unique identifier for the session/connection generated by Zeek. | Stable key for joining across logs (e.g., conn.log to http.log), deduplication, and traceability. |
| id.orig_h | Originator host IP (source IP) of the flow. | Identifies which host initiated communication; crucial for attribution, lateral movement tracing, and policy enforcement. |
| id.orig_p | Originator port (source port). | Helps discriminate flow types and detect anomalous source port behavior (e.g., malware using unusual ephemeral ports). |
| id.resp_h | Responder host IP (destination IP). | Reveals the remote peer; useful for reputation checks, allow/deny lists, and IOC matching. |
| id.resp_p | Responder port (destination port). | Indicates targeted service (e.g., 80/443/53); supports protocol inference and rule-based alerts. |
| proto | Transport protocol (e.g., tcp, udp, icmp). | Splits analysis paths (e.g., TCP state vs. UDP stateless), drives parsers, and informs detection signatures. |
| service | Application service detected by Zeek (e.g., http, dns, ssl). May be empty if undetected. | Provides higher-level context beyond ports; drives content-specific analysis and feature extraction. |
| duration | Length of the connection in seconds. For UDP/ICMP may be near-zero. | Behavioral signal (long-lived vs. short bursts), helps find beacons, tunnels, and stalled connections. |
| orig_bytes | Total payload bytes sent by the originator. | Directional volume feature for traffic profiling, anomaly detection, and data exfil estimation. |
| resp_bytes | Total payload bytes sent by the responder. | Complements `orig_bytes` to understand flow symmetry/asymmetry; useful for C2 patterns and failed handshakes. |
| conn_state | Zeek’s symbolic connection state (e.g., S0, S1, SF, REJ). Encodes handshake/results for TCP; limited for UDP. | Flags incomplete, failed, or reset connections; key for scanning detection and reliability scoring. |
| local_orig | Boolean/flag indicating the originator is local (per Zeek’s Site::local_nets). | Distinguishes internal-to-external vs. external-to-internal flows; supports perimeter and egress policy analysis. |
| local_resp | Boolean/flag indicating the responder is local. | Identifies inbound to internal services; helps focus detection on exposed assets and unsolicited inbound traffic. |
| missed_bytes | Number of bytes Zeek knows were missed due to packet loss or truncation. | Data quality indicator; alerts analysts to incomplete evidence and reduces false conclusions. |
| history | Compact string summarizing observed TCP events (e.g., `S`=SYN, `H`=server SYN-ACK, `D`=data, `F`=FIN, `R`=RST). For UDP, may include `X` (connection attempt) etc. | Rich per-flow handshake/behavior fingerprint; helps detect scans, resets, and odd sequences. |
| orig_pkts | Count of packets sent by the originator. | Directional packet-level volume used in feature engineering; helps detect floods and anomalies. |
| orig_ip_bytes | Total IP-layer bytes sent by the originator (includes headers). | Complements `orig_bytes` (payload) to gauge overhead; useful for MTU/fragmentation and protocol peculiarities. |
| resp_pkts | Count of packets sent by the responder. | Paired with `orig_pkts`; helps identify asymmetric flows, retries, and server-driven chatter. |
| resp_ip_bytes | Total IP-layer bytes sent by the responder. | Directional size insight; supports detection of large server responses or minimal ACK-only patterns. |
| tunnel_parents | List/identifier of parent tunnel UIDs if the flow is inside a tunnel (e.g., VPN, GRE); may be empty. | Maintains lineage for encapsulated flows; critical for understanding overlay networks and nested sessions. |
| label | High-level label for the flow (e.g., Benign or Malicious). Provided by CTU dataset curation. | Enables supervised learning, evaluation, and quick filtering during exploratory analysis. |
| detailed-label | Fine-grained label with additional context (e.g., malware family, botnet type, or scenario). | Supports deeper ML tasks (multi-class), IOC mapping, and more nuanced evaluations beyond binary labels. |

Additional notes
- Field presence: Some fields can be empty depending on protocol or capture conditions (e.g., `service`, `duration`, `tunnel_parents`).
- Units: Durations are in seconds; bytes are counts; timestamps are epoch seconds (float or integer depending on export).
- Joins: Use `uid` to join with other Zeek logs (e.g., dns.log, http.log) if available.
- Local nets: `local_orig` and `local_resp` depend on sensor configuration; in some datasets these may be inferred or defaulted.
